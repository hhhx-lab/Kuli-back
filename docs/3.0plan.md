# 酷里 3.0 正式发布升级计划

## Summary

酷里 3.0 的目标是把当前“官网 + 订单系统 + 小酷雏形”升级为更适合公开发布的服务平台。核心升级包括：前后端目录进一步分离、邮箱注册登录与登录门禁、文档中心式知识库、个人主页与积分邀请、公开子产品导航页、页面视觉差异化，以及真正可接入 GPT-5.5 和本地 Markdown 知识库的小酷 Agent。

3.0 不是重做一个营销页，而是把用户从“第一次进站了解服务”到“注册、写小纸条、查看订单、咨询小酷、阅读文档、邀请朋友”的完整闭环做成熟。实现时要继续保持权限边界：游客只能看公开内容，普通用户只能看自己的订单和个人数据，管理员才能看全部订单和内部字段。

为了达到正式发布标准，3.0 还必须补齐账号安全、邮件与站内通知、隐私条款、SEO、运维监控、文档治理和分期交付。管理员回复邮件通知对应能力按现有源码整理目录并接入统一通知体系，不在 3.0 中重复设计为全新功能。

## 当前仓库事实与目录基线

- 当前仓库已经按 `apps/web` 与 `apps/api` 组织新前后端；旧 `frontend/`、`backend/` 已从工作树删除，根目录保留聚合脚本、文档、合同和部署文件。参考：`package.json`、`apps/web/`、`apps/api/`。
- API 当前使用 FastAPI 入口集中在 `apps/api/app/main.py`，邮箱登录和注册接口已存在：`apps/api/app/main.py:454`、`apps/api/app/main.py:462`。
- 用户模型已经包含邮箱验证、失败登录次数、锁定时间、积分与邀请码字段，适合作为 3.0 账号安全和个人主页的扩展基线：`apps/api/app/models/entities.py:42`。
- 订单权限基线已经存在：用户订单列表只读取当前用户订单，用户订单详情按 `owner_user_id` 过滤，管理员订单接口单独走管理员依赖：`apps/api/app/main.py:555`、`apps/api/app/main.py:567`、`apps/api/app/main.py:739`。
- 管理员消息入口已存在，当前在发送消息时写入 `order_messages` 和 `admin_audit_logs`，并触发订单自动化：`apps/api/app/main.py:840`。3.0 的管理员回复邮件通知目录整理应围绕这个入口接入事件，而不是新增并行的回复入口。
- 通知事件数据基础已存在：`NotificationEvent` 模型在 `apps/api/app/models/entities.py:341`，Alembic 初始迁移中有 `notification_events` 表：`apps/api/alembic/versions/0001_kuli_v2_schema.py:241`。当前尚未形成独立 `app/notifications/` 包，因此 3.0 要把现有通知相关能力和后续邮件 provider 统一迁入该目录。
- 后台任务已有 Celery 组织方式，当前 include 附件、知识 embedding 和订单自动化任务：`apps/api/app/tasks/celery_app.py:7`。通知发送任务应延续这个目录和注册方式。
- 文档中心源码已读取固定五个 Markdown 文档，并解析 frontmatter、锚点、搜索：`apps/api/app/services/docs.py:8`、`apps/api/app/services/docs.py:41`、`apps/api/app/services/docs.py:88`、`apps/api/app/services/docs.py:124`。
- 知识索引脚本已具备读取文档、切 chunk、写入 article/chunk/embedding 的基础流程：`scripts/index_knowledge.py:18`。
- 当前仓库没有 `openspec/specs/` 基线目录；已有 OpenSpec change 为 `add-kuli-order-portal`，`openspec list` 显示已完成，后续 3.0 应创建独立 change，避免和 1.0/2.0 变更混在一起。

## 本轮补充的 7 个发布级要求

这 7 点是 3.0 从“功能迭代版”走向“可公开发布版”的质量补充，后续 OpenSpec 拆解时必须作为独立验收边界处理。

### 1. 账号安全与反滥用

- 目标：注册、登录、密码、限流、小酷调用和写接口都有服务端防护，避免垃圾注册、撞库、弱密码、脚本提交和 Agent 滥用。
- 影响范围：`/api/auth/*`、`/api/orders`、`/api/uploads/*`、`/api/agent/*`、管理员接口、审计日志和前端错误提示。
- 关键规则：
  - 密码至少 8 位，包含字母和数字，不能等于邮箱、空值或常见弱密码。
  - 登录失败不暴露邮箱是否存在；同邮箱和同 IP 连续失败进入冷却。
  - 注册、写小纸条、上传、小酷对话、密码重置都要有请求频率、请求体大小和异常行为限制。
  - 小酷限流按游客 visitorId/IP、登录 userId、管理员 userId 分层。
  - 管理员接口必须由服务端 role 校验，不能只靠前端隐藏路由。
- 验收：API 单测覆盖弱密码、重复邮箱、失败登录、锁定、游客写接口拦截、普通用户访问管理员接口、Agent 限流与拒答日志。

### 2. 邮件、站内通知与目录整理

- 目标：统一通知事件、邮件发送、站内通知和模板目录，让订单消息、管理员回复、报价、交付、验收、邮箱验证、密码重置都走同一条事件链路。
- 重要边界：管理员回复邮件通知按源码已有能力整理目录，不作为 3.0 从零重写的新功能；实施时先定位现有回复通知入口，补行为测试，再迁入统一目录。
- 已验证基线：
  - `NotificationEvent` 模型已存在：`apps/api/app/models/entities.py:341`。
  - `notification_events` 迁移表已存在：`apps/api/alembic/versions/0001_kuli_v2_schema.py:241`。
  - 管理员公开/内部消息入口已存在：`apps/api/app/main.py:840`。
- 目标目录：
  - `apps/api/app/notifications/__init__.py`
  - `apps/api/app/notifications/events.py`：通知事件类型、幂等 key、事件创建与状态流转。
  - `apps/api/app/notifications/email.py`：邮件 provider 抽象、发送入口、失败处理，不直接暴露 SMTP/API key。
  - `apps/api/app/notifications/in_app.py`：用户可见站内通知创建、已读状态、订单跳转链接。
  - `apps/api/app/notifications/templates/`：邮件模板目录。
  - `apps/api/app/notifications/templates/admin_reply.html`
  - `apps/api/app/notifications/templates/email_verify.html`
  - `apps/api/app/notifications/templates/password_reset.html`
  - `apps/api/app/tasks/notification_tasks.py`：Celery/RQ 发送通知、重试、失败落库。
  - `apps/api/tests/test_notifications.py`：覆盖管理员回复、邮件失败、站内通知兜底、权限隔离。
- 验收：管理员发送客户可见回复后创建通知事件和站内通知；邮件发送由 worker 消费事件；邮件失败时事件记录错误与重试次数，站内通知仍可见；内部备注不触发普通用户通知。

### 3. 发布、运维、监控与回滚

- 目标：把 local / staging / production 的启动、依赖、健康检查和回滚边界写清楚，避免“本地能跑”直接被当成生产可发布。
- 影响范围：`.env.example`、Docker/compose、Alembic、对象存储、Redis、Celery worker、知识索引、前端部署和 README。
- 必须补齐：
  - `GET /api/health` 和 `GET /api/health/deps`。
  - 数据库、Redis、对象存储、邮件 provider、LLM/RAG 的依赖检查。
  - API 请求日志、认证安全日志、管理员审计日志、小酷调用日志、通知发送日志。
  - Postgres 备份、对象存储生命周期、知识索引失败降级、前端上一版本回滚。
- 验收：staging 能跑 Postgres + Redis + 对象存储 + 邮件 sandbox；生产 env 不使用默认 secret；Alembic、前端和知识索引有可执行的回滚或降级说明。

### 4. 隐私、服务条款与上传内容说明

- 目标：正式发布前让用户知道酷里收集什么、如何处理订单附件、小酷对话有什么边界、哪些敏感信息不能上传。
- 影响范围：页脚入口、法律页面、上传提示、小酷拒答、注册/小纸条提交前提示。
- 页面：
  - `/legal/privacy`
  - `/legal/terms`
  - `/legal/upload-policy`
- 必须说明：
  - 收集邮箱、订单内容、附件 metadata、小酷对话、访问日志的用途。
  - 不出售个人数据，不向其他用户展示订单或附件。
  - 密码、验证码、私钥、支付凭证、身份证等敏感信息不要直接提交给小酷或小纸条。
  - 小酷不代表最终报价、周期或管理员承诺。
- 验收：三页可访问，页脚可进入；上传区和小酷敏感信息场景有一致提醒；浏览器 smoke 覆盖法律页面。

### 5. 通知中心

- 目标：用户登录后能集中看到订单消息、管理员回复、报价、交付、验收、系统消息和邀请积分，不依赖邮件单点成功。
- 影响范围：`notifications` 数据表、`/api/notifications*`、右上角账号菜单、`/me`、订单详情页。
- 功能边界：
  - 普通用户只能看到自己的通知。
  - 管理员内部通知不下发给普通用户。
  - 订单类通知可跳转到订单详情。
  - 支持未读数量、列表、单条已读、全部已读。
  - 邮件失败不影响站内通知生成。
- 验收：用户收到管理员公开回复、报价、交付、验收提醒；通知角标正确；标记已读后未读数下降；其他用户无法读取该通知。

### 6. 文档中心内容治理

- 目标：让文档中心、服务页、小酷知识库使用同一份稳定 Markdown 来源，避免不同页面说法不一致。
- 影响范围：`apps/api/knowledge/docs/`、`GET /api/docs*`、`scripts/index_knowledge.py`、`scripts/knowledge_doctor.py`、小酷 citations。
- 维护规则：
  - 每个文档必须有 `slug`、`title`、`description`、`tags`、`order`、`updated_at`、`status`。
  - 发布后的 slug 不随意修改；必须修改时提供 alias 或 redirect。
  - 涉及价格、周期、成功率时使用谨慎措辞，以管理员确认为准。
  - 文档不得要求用户发送密码、验证码、私钥或支付凭证。
  - 每次修改文档后必须运行知识索引和 doctor。
- 验收：五个文档能阅读、搜索、被小酷引用；doctor 能发现缺 frontmatter、slug 重复、空文档和索引缺失；citations 能回到具体文档锚点。

### 7. SEO 与公开增长

- 目标：让首页、服务详情、文档中心和产品页具备正式发布后的搜索与分享基础。
- 影响范围：Nuxt metadata、公开路由、文档内容、服务详情页、产品页、静态生成或服务端输出。
- 必须补齐：
  - 首页、服务详情、文档页、产品页 title / description。
  - `sitemap.xml`、`robots.txt`、canonical URL。
  - Open Graph title / description / image。
  - FAQ structured data，优先覆盖 FAQ 文档和服务详情页。
  - 分享图和公开页面文案不能出现本地 demo、测试账号、内部成本等内容。
- 验收：公开页面源码含 metadata；sitemap/robots 可访问；服务详情和 FAQ 有结构化数据；浏览器 smoke 覆盖 sitemap、robots 和关键公开路由。

## Key Changes

### 1. 前后端目录进一步分离

- 保留同一个 Git 仓库，但让 `apps/web` 和 `apps/api` 成为更独立的两个应用。
- `apps/web` 拥有自己的 `.env.example`、README、启动说明、构建说明和部署说明。
- `apps/api` 拥有自己的 `.env.example`、README、启动说明、迁移说明、知识索引说明和 worker 说明。
- 根目录只保留聚合脚本、合同生成、Docker 编排、总 README 和 docs，不让前端直接依赖后端源码，也不让后端依赖前端构建产物。
- 根目录脚本建议调整为：
  - `npm run dev:all`：同时启动 API 和 Web。
  - `npm run dev:web`：只启动 Nuxt。
  - `npm run dev:api`：只启动 FastAPI。
  - `npm run test:all`：聚合测试。
  - `npm run contracts:generate` / `contracts:check`：同步 OpenAPI 合同。
- 分离后必须保证当前网站能正常运行，所有已有核心流程不能回退。

### 2. 邮箱注册登录与访问门禁

- 登录继续使用邮箱 + 密码。
- 注册使用邮箱 + 密码 + 展示名，公开注册只能创建普通账号。
- 管理员账号不能通过公开注册创建，只能通过种子数据、后台配置或后续单独的管理员创建流程产生。
- 游客只能访问：
  - `/`
  - `/services`
  - `/services/:slug`
  - `/help`
  - `/help/:slug`
  - `/products`
  - `/login`
- 登录后才能访问：
  - `/note`
  - `/orders`
  - `/orders/:orderNumber`
  - `/me`
  - `/settings`
  - `/referrals`
- 管理员才能访问：
  - `/admin`
  - `/admin/orders/:orderNumber`
- 游客点击“写小纸条”“我的订单”“个人主页”等受保护入口时，跳转到 `/login?redirect=<原目标>`。
- 登录或注册成功后回到 redirect 页面。
- 普通用户访问管理员页面时不展示内部信息，应显示无权限状态或跳回 `/orders`。
- 订单创建主流程改为必须登录后调用 `POST /api/orders`，订单必须绑定 `owner_user_id`。
- 旧的游客询单入口可以保留为兼容 API，但不作为前端主流程使用；正式发布版前端不再让游客直接提交订单。

### 3. 个人主页、账号菜单、积分与邀请

- 右上角登录态入口从单个 CTA 升级为账号菜单。
- 登录后右上角展示用户邮箱或展示名，点击后展开菜单：
  - 个人主页
  - 我的订单
  - 设置
  - 积分进度
  - 邀请注册
  - 退出登录
- 新增 `/me` 个人主页：
  - 显示登录邮箱、展示名、角色、注册时间。
  - 显示我的订单摘要：总订单数、进行中、待补充、待验收、已完成。
  - 显示最近订单列表，支持跳转到订单详情。
  - 显示积分进度：当前积分、下一等级进度、获得积分方式。
  - 显示邀请注册卡片：邀请码、邀请链接、复制按钮、邀请成功说明。
- 新增 `/settings` 设置页：
  - 修改展示名。
  - 查看邮箱。
  - 退出登录。
  - 第一版不做邮箱更换、密码找回、二次验证、头像上传。
- 新增 `/referrals` 或在 `/me` 中内嵌邀请模块：
  - 用户拥有唯一邀请码。
  - 新用户通过邀请链接注册后，邀请人获得积分。
  - 第一版只记录邀请注册积分，不做积分商城、不做支付抵扣。
- 后端补充用户接口：
  - `GET /api/me/profile`
  - `PATCH /api/me/profile`
  - `GET /api/me/summary`
  - `GET /api/me/referral`
- 数据模型补充：
  - 用户积分字段。
  - 邀请码字段。
  - 邀请人字段。
  - 邀请奖励记录表或 audit log。

### 4. 文档中心式知识库

- 将当前 `/help` 从知识卡片列表升级为正式文档中心，参考火山引擎文档中心这类结构。
- 文档中心不是普通 FAQ 页，而是用户可以阅读、搜索、跳转、咨询小酷的长期知识入口。
- 文档中心布局：
  - 顶部：文档中心标题、搜索框、问小酷入口。
  - 左侧：文档目录树。
  - 中间：正文阅读区。
  - 右侧：当前文档锚点目录、相关文档、联系入口。
  - 底部：反馈“这篇有帮助吗”、联系我们、继续写小纸条。
- 第一版固定五个一级文档：
  - `快速开始`：从了解酷里、注册、写小纸条、查看订单到验收的最短路径。
  - `核心概念`：小纸条、服务分类、订单状态、定金/尾款、验收、交付物、小酷 Agent、知识库。
  - `常见问题`：收费、材料、账号安全、订单进度、售后边界、哪些需求不接。
  - `指南`：如何描述需求、如何上传材料、如何看报价、如何验收、如何和管理员沟通。
  - `联系我们`：咨询方式、人工介入边界、敏感信息提醒、服务时间和响应预期。
- 文档源存放在本地 Markdown 目录：
  - `apps/api/knowledge/docs/quick-start.md`
  - `apps/api/knowledge/docs/concepts.md`
  - `apps/api/knowledge/docs/faq.md`
  - `apps/api/knowledge/docs/guides.md`
  - `apps/api/knowledge/docs/contact.md`
- 每个 Markdown 文件使用 frontmatter：

```yaml
---
slug: quick-start
title: 快速开始
category: docs
description: 从第一次访问酷里到提交小纸条的最短路径。
tags: [入门, 小纸条, 订单]
order: 1
updated_at: 2026-05-30
---
```

- 前端文档路由：
  - `/help` 默认打开快速开始。
  - `/help/:slug` 打开指定文档。
- API 补充：
  - `GET /api/docs` 返回目录、slug、标题、描述、标签、更新时间。
  - `GET /api/docs/:slug` 返回正文、HTML/Markdown 内容、锚点、相关文档。
  - `GET /api/docs/search?q=...` 搜索文档内容。
- 文档中心与小酷使用同一份 Markdown 知识源，避免页面说法和 Agent 回答不一致。

### 5. OpenViking 式小酷知识索引

- 小酷知识接入参考 OpenViking 的 agent setup 思路：配置明确、索引可复跑、健康可检查、失败可降级。
- 新增知识索引命令：

```bash
PYTHONPATH=apps/api uv run --project apps/api python scripts/index_knowledge.py
```

- `index_knowledge.py` 职责：
  - 读取 `KNOWLEDGE_ROOT` 下 Markdown。
  - 解析 frontmatter、标题层级、段落、列表和锚点。
  - 按标题和语义段落切 chunk。
  - 写入 `knowledge_articles`、`knowledge_chunks`、`knowledge_embeddings`。
  - 为每个 chunk 保存 `source_path`、`slug`、`section`、`anchor`、`priority`、`updated_at`。
- 新增知识健康检查命令：

```bash
PYTHONPATH=apps/api uv run --project apps/api python scripts/knowledge_doctor.py
```

- `knowledge_doctor.py` 职责：
  - 检查五个必需文档是否存在。
  - 检查 frontmatter 是否完整。
  - 检查文档是否为空。
  - 检查 slug 是否重复。
  - 检查 chunk 数量是否合理。
  - 检查 embedding 维度是否符合 `VECTOR_DIMENSION`。
  - 检查 `OPENAI_API_KEY`、`OPENAI_MODEL`、`EMBEDDING_MODEL` 是否配置。
  - 输出小酷当前处于 `remote-rag`、`local-rag` 或 `local-rules-fallback`。
- 小酷检索流程：
  - 读取用户消息。
  - 读取当前页面上下文：路由、服务 slug、文档 slug、是否登录。
  - 登录后只读取当前用户自己的订单摘要。
  - 从文档中心、服务目录、FAQ、订单规则中检索 3-5 个高相关 chunk。
  - 将 chunk 引用注入 GPT-5.5 prompt。
  - 生成短回答、行动按钮、可选小纸条草稿和 citations。
- 小酷回答必须带边界：
  - 只回答酷里服务、订单、材料、付款、验收、文档中心相关问题。
  - 不回答通用闲聊、无关知识、违法违规、绕过安全限制、破解账号等问题。
  - 不承诺价格、周期、成功率、账号开通结果。
  - 不要求用户提供密码、验证码、私钥、支付凭证。
  - 不读取其他用户订单。
  - 不展示成本、利润、管理员备注、内部优先级。
- 小酷响应建议结构：

```json
{
  "answer": "短回答",
  "actions": [
    { "label": "查看快速开始", "to": "/help/quick-start" },
    { "label": "写小纸条", "to": "/note" }
  ],
  "citations": [
    { "title": "快速开始", "to": "/help/quick-start#write-note" }
  ],
  "draft": {
    "serviceSlug": "not-sure",
    "summary": "用户想先判断需求是否能做",
    "missingFields": ["截止时间", "参考材料"]
  }
}
```

### 6. GPT-5.5 与小酷提示词

- 小酷使用 env 配置模型，不在代码里写死 key。
- `.env.example` 必须准备好以下字段：

```dotenv
OPENAI_API_KEY=
OPENAI_MODEL=gpt-5.5
OPENAI_BASE_URL=https://api.openai.com/v1
EMBEDDING_MODEL=text-embedding-3-small
KNOWLEDGE_ROOT=apps/api/knowledge/docs
XIAOKU_AGENT_ENABLED=true
XIAOKU_REQUIRE_CITATIONS=true
LLM_TIMEOUT_SECONDS=20
VECTOR_DIMENSION=1536
```

- 小酷系统提示词必须覆盖：
  - 角色：酷里的 3D 小猫服务助手。
  - 目标：帮助用户理解服务、整理需求、查看自己的订单进度。
  - 语气：机灵、轻松、靠谱，不装专家。
  - 范围：只回答酷里业务和知识库相关内容。
  - 拒答：无关问题、敏感信息、违法违规、内部字段、其他用户订单。
  - 价格边界：不直接报价，只说明需要管理员确认。
  - 成功率边界：不保证第三方平台结果。
  - 输出：短回答 + 2-4 个行动按钮 + 必要 citations。
  - 缺知识：如果检索上下文不足，应说明“这个我现在没有足够资料”，并引导写小纸条或联系管理员。
- 小酷第一版不直接调用创建订单工具，只生成草稿并引导用户确认提交。

### 8. 页面视觉与组件差异化

- 3.0 要避免每个页面都像同一个 hero + card + grid 模板。

- 页面设计方向：
  - 首页：品牌入口 + 服务横向带 + 流程分段 + 信任内容 + 小酷引导。

  - 文档中心：技术文档布局，左侧目录、中间正文、右侧锚点，不使用大面积卡片堆叠。
  - 产品页：产品导航布局，类似工具库/产品目录，而不是订单工作台。
  - 小纸条页：表单向导布局，强调低压力填写和小酷协助。
  - 我的订单：工作台布局，突出进度、下一步、沟通和附件。
  - 个人主页：账户仪表盘布局，突出邮箱、订单摘要、积分和邀请。
  - 管理后台：密集操作台布局，强调搜索、筛选、AI 建议、内部字段和操作效率。
- 可参考方向：
  - Claude Code 官网的克制排版和技术感。
  - 大厂文档中心的信息架构。
  - 云厂商文档的目录和正文阅读体验。
  - 产品官网的分区节奏。
- 约束：
  - 不堆纯装饰渐变。
  - 不用千篇一律的重复卡片。
  - 移动端不能横向溢出。
  - 小酷不能遮挡主 CTA、表单提交、付款说明、上传入口。

### 9. 订单与权限边界

- 普通用户只能看到：
  - 自己的订单。
  - 自己订单的公开消息。
  - 自己订单的公开附件。
  - 报价、付款记录、交付物和公开备注。
- 普通用户不能看到：
  - 其他用户订单。
  - 成本。
  - 利润。
  - 管理员内部备注。
  - 内部优先级。
  - 内部分工。
- 管理员可以看到全部订单和内部字段。
- 小酷即使在管理员页面，也不能替管理员自动发送报价或自动承诺结果。
- 小酷对普通用户只能读取当前用户自己的订单摘要。
- 邀请积分不能泄露被邀请人的敏感信息，只展示邀请数量和奖励记录摘要。

### 10. 账号安全与反滥用

- 公开注册必须增加基础防刷策略，避免正式发布后被垃圾账号、脚本提交和小酷滥用拖垮。
- 密码规则：
  - 最少 8 位。
  - 必须包含字母和数字。
  - 禁止空密码、邮箱本身、常见弱密码。
- 登录失败限制：
  - 同一邮箱连续失败 5 次后短时间锁定。
  - 同一 IP 高频登录失败进入冷却。
  - 错误提示不暴露“邮箱是否存在”，统一提示“邮箱或密码不正确”。
- 注册限制：
  - 同一 IP 短时间注册数量限制。
  - 邀请注册奖励进入可疑状态时不立即发放积分。
  - 对一次性邮箱域名可以先记录风险，不一定第一版强拦截。
- 小酷限流：
  - 游客按 IP / visitorId 限制每日消息数。
  - 登录用户按 userId 限制每日消息数。
  - 管理员不共享普通用户额度，但也要有防误用上限。
- API 防护：
  - 所有写接口需要服务端鉴权。
  - 管理员接口必须检查 role，不能只依赖前端路由。
  - 上传、消息、小酷 chat、注册、登录必须有请求体大小限制。
- 审计：
  - 登录失败、注册、密码重置、管理员状态修改、积分发放、邀请奖励、小酷拒答都进入 audit 或 security log。

### 11. 邮件、站内通知与现有管理员回复通知目录整理

- 3.0 需要建立统一通知体系，服务注册验证、密码找回、订单状态变化、报价、交付、验收、邀请积分等场景。
- 邮件能力分层：
  - 邮箱验证：注册后发送验证链接。
  - 密码重置：用户忘记密码时发送一次性重置链接。
  - 系统通知邮件：订单状态变化、报价、交付、验收提醒等。
  - 管理员回复邮件通知：该能力按现有源码整理目录并接入统一通知事件，不作为 3.0 从零重写的新能力。
- 目录整理目标：
  - `apps/api/app/notifications/__init__.py`
  - `apps/api/app/notifications/events.py`：通知事件类型、创建事件、幂等 key。
  - `apps/api/app/notifications/email.py`：邮件 provider、发送入口、失败处理。
  - `apps/api/app/notifications/in_app.py`：站内通知创建、已读状态、订单跳转链接。
  - `apps/api/app/notifications/templates/`：邮件模板目录。
  - `apps/api/app/notifications/templates/admin_reply.html`
  - `apps/api/app/notifications/templates/email_verify.html`
  - `apps/api/app/notifications/templates/password_reset.html`
  - `apps/api/app/tasks/notification_tasks.py`：Celery/RQ worker 发送通知。
- 管理员回复通知整理方式：
  - 以当前管理员消息入口 `POST /api/admin/orders/{order_number}/messages` 为触发点。
  - 管理员发送客户可见消息时创建 `notification_events` 和用户站内通知。
  - 如果现有源码中已有独立邮件发送逻辑，先补行为测试，再迁入 `apps/api/app/notifications/email.py`；如果当前只存在通知事件模型，则只在统一通知目录内补齐 provider，不创建第二套并行实现。
  - 通知任务读取事件后发送邮件。
  - 发送成功后更新状态为 `sent`。
  - 发送失败后记录错误和重试次数。
  - 不把 SMTP/API key 写进代码，只走 env。
- 站内通知：
  - 所有邮件通知对应生成一条站内通知，避免邮箱失败导致用户完全收不到提醒。
  - 用户个人主页显示未读通知数量。
  - 订单详情页显示与该订单相关的通知。
- 邮件 env：

```dotenv

MAIL_REPLY_TO=

SMTP_PORT=
SMTP_USERNAME=
SMTP_PASSWORD=
EMAIL_VERIFY_TOKEN_EXPIRE_MINUTES=60
PASSWORD_RESET_TOKEN_EXPIRE_MINUTES=30
```

## Public APIs / Interfaces

### Auth

- `POST /api/auth/register`
  - 输入：`email`、`password`、`displayName`、可选 `referralCode`。
  - 输出：token 和普通用户信息。
  - 不接受 `role`。
- `POST /api/auth/login`
  - 输入：`email`、`password`。
  - 输出：token 和用户信息。
- `GET /api/auth/me`
  - 输出：当前用户基础信息。
- `POST /api/auth/email-verification/request`
  - 登录后请求发送邮箱验证邮件。
- `POST /api/auth/email-verification/confirm`
  - 输入验证 token，确认邮箱。
- `POST /api/auth/password-reset/request`
  - 输入邮箱；无论邮箱是否存在都返回相同提示。
- `POST /api/auth/password-reset/confirm`
  - 输入 reset token 和新密码。

### User Center

- `GET /api/me/profile`
  - 返回邮箱、展示名、角色、注册时间、积分、邀请码。
- `PATCH /api/me/profile`
  - 第一版只允许修改展示名。
- `GET /api/me/summary`
  - 返回订单数量、状态统计、最近订单、积分进度。
- `GET /api/me/referral`
  - 返回邀请码、邀请链接、邀请人数、积分奖励摘要。

### Docs / Knowledge

- `GET /api/docs`
  - 返回五个文档的目录信息。
- `GET /api/docs/:slug`
  - 返回指定文档正文、锚点、更新时间、相关文档。
- `GET /api/docs/search?q=...`
  - 返回匹配文档和片段。
- `GET /api/knowledge` 与 `GET /api/knowledge/search`
  - 保留，作为小酷和兼容旧知识列表的底层检索接口。

### Notifications

- `GET /api/notifications`
  - 返回当前用户站内通知列表，支持 `status=unread/read`。
- `PATCH /api/notifications/:id/read`
  - 标记单条通知已读。
- `PATCH /api/notifications/read-all`
  - 标记当前用户全部通知已读。
- `GET /api/notifications/unread-count`
  - 返回右上角菜单显示的未读数量。
- 管理员回复邮件通知不新增单独公开 API；管理员发送公开消息时由后端创建通知事件，再由通知任务发送邮件。

### Orders

- `POST /api/orders`
  - 必须登录。
  - 创建归属于当前用户的订单。
- `GET /api/orders`
  - 返回当前用户订单。
- `GET /api/orders/:orderNumber`
  - 只能返回当前用户自己的订单。
- 管理员订单接口保持管理员权限。

### Agent

- `POST /api/agent/sessions`
  - 输入：`pagePath`、可选 `visitorId`、可选 `docSlug`、可选 `serviceSlug`。
- `POST /api/agent/chat`
  - 输入：`sessionId`、`message`、可选页面上下文。
  - 输出：`answer`、`actions`、可选 `citations`、可选 `draft`。

### 1. API 单测

- 注册新邮箱成功。
- 重复邮箱返回 409。
- 注册请求传入 `role=admin` 不会创建管理员。
- 密码过短或弱密码返回 422。

- 同一个被邀请用户不会重复给邀请人加分。
- 邀请异常场景不立即发放积分，进入待审核或不发放状态。
- 未登录访问 `/api/orders` 返回 401。

- 普通用户只能读取自己的站内通知。

- `/api/docs` 返回五个文档。

## Assumptions

- 3.0 继续采用 Nuxt / Vue 3 / TypeScript + FastAPI / Python。
- 采用同仓库目录分离，不拆成两个 Git 仓库。
- `/products` 是公开产品导航页，不接真实项目管理。
- 文档中心第一版只维护五个固定 Markdown 文档，不做在线 CMS。
- 积分第一版只用于邀请注册和成长展示，不做支付抵扣。
- 小酷第一版不直接创建订单，只生成草稿和引导按钮。
- GPT-5.5 通过 env 配置，可替换为 OpenAI-compatible base URL。
- 无 API key 时必须保留本地规则 fallback，保证网站可启动、可演示、可测试。
- 管理员回复邮件通知已有对应源码时，3.0 只做目录整理、事件接入、测试补齐，不重新发明一套并行实现。
- 邮件 provider 第一版允许使用 SMTP 或兼容邮件 API，但必须通过统一 `notifications` 服务调用。
- 页面设计参考大厂文档中心和 Claude Code 官网的成熟感，但最终视觉必须保留酷里自己的小窗口、3D 小猫和亲近服务气质。
