# Kuli 2.0 订单自动化与小酷 Agent 设计文档

## 1. 背景

Kuli 1.0 已经完成官网、服务详情、小纸条、附件 metadata、邮箱密码登录、普通用户订单工作台、管理员订单管理台、人工报价/付款/交付物记录，以及一个基于规则的 agent brief。当前 1.0 的核心价值是“能提交、能查看、能管理”，但客户体验还偏静态，管理员需要手动判断和维护订单进度，后台检索和自动化能力也不够成熟。

2.0 的方向是把订单系统从“记录工具”升级为“半自动服务台”：客户提交问题后，系统能主动引导、补全、解释服务内容；管理员收到订单后，AI 能读取原话和附件信息、生成结构化判断、建议下一步状态和沟通话术；用户侧通过“小酷”虚拟人物获得更有趣、更亲近的陪伴式体验。

## 2. 设计目标

1. 让客户感受到“有问题都可以先来问问”，而不是被固定服务分类卡死。
2. 让服务说明更可信：每个服务可以展示能力清单、案例、材料要求、典型流程和不收费咨询边界。
3. 让管理员后台更像真实订单工作台：搜索、筛选、自动摘要、AI 建议状态、待办提醒、客户沟通建议和进度维护都在一个地方完成。
4. 让订单进度尽可能自动维护：AI 和规则引擎根据客户动作、管理员动作、附件、报价、付款、交付物、验收反馈生成状态建议；低风险节点可自动更新，高风险节点需管理员确认。
5. 在用户页面加入虚拟人物 Agent “小酷”：能追随鼠标、做有趣动作、解释服务、回答常见问题、引导写小纸条、提醒订单状态。
6. 小酷的知识库覆盖 Kuli 全部业务内容，并且与官网服务目录、FAQ、结算规则、风险边界保持一致。

## 3. 非目标

1. 2.0 不直接接在线支付网关；仍以人工收款记录为主。
2. 2.0 不让 AI 自动承诺价格、周期、账号开通成功率或第三方平台结果。
3. 2.0 不让 AI 自动读取用户上传文件全文并对外发送结果，除非后续单独加入文件解析权限、大小限制和隐私提示。
4. 2.0 不把小酷做成干扰用户操作的装饰物；它必须可关闭、可静音、可减少动画。
5. 2.0 不默认把所有 AI 建议自动写入订单；涉及报价、取消、完成、退款、敏感风险的动作必须管理员确认。

## 4. 推荐方案

### 方案 A：规则优先 + AI 辅助（推荐）

订单状态由规则引擎保障确定性，AI 负责解释、摘要、推荐下一步和生成话术。比如用户提交小纸条自动进入 `submitted`，管理员提问进入 `clarifying`，管理员报价进入 `quoted`，交付物上传进入 `review`。AI 可以建议“该订单可能需要追问文件页数”，但不直接改高风险字段。

优点：可控、易测试、不会因为模型输出不稳定破坏订单状态。  
缺点：自动化看起来没那么“魔法”，需要把规则设计清楚。

### 方案 B：AI 代理主导订单状态

AI 读取订单事件流后直接决定状态、待办和客户消息草稿，管理员只做审核。  
优点：自动化感强，后台更轻。  
缺点：风险高，容易误判收费边界、交付节点或客户意图。

### 方案 C：纯手动后台 + 小酷只做客服

只增强前台小酷，后台维持手动管理。  
优点：实现快。  
缺点：无法解决你提到的“管理订单不够成熟、自动化不够”的核心问题。

推荐采用方案 A：规则优先 + AI 辅助。这样 2.0 能快速提高体验，同时不会让订单系统失控。

## 5. 2.0 功能范围

### 5.1 服务内容与知识库

把当前 `serviceCatalog` 从页面数据升级为“服务知识库源”。知识库不只服务前台展示，也服务小酷问答、需求润色、订单摘要和管理员判断。

知识库每个服务包含：

- 服务定位：一句话说明可以帮什么。
- 典型烦恼：用户自然语言场景，如“PDF 翻译后格式乱了”“Claude Code 授权跑不通”。
- 可交付物：文件、配置说明、最小 demo、排查结论、部署结果、源码、操作说明。
- 用户需准备材料：截图、源文件、账号状态、报错日志、截止时间、预算、可远程程度。
- 风险边界：不能保证第三方平台结果、低质量扫描件限制、不接违规需求。
- 典型流程：先判断、补材料、报价、处理、交付、验收。
- 咨询/收费规则：哪些可以先免费判断，哪些进入正式订单。
- 小酷回答素材：适合对话展示的短句版本。
- 管理员判断提示：建议追问、建议状态、常见报价因子。

前端页面增加：

- `/services`：服务总览，突出“你不用分得很准，先讲烦恼”。
- `/services/:slug`：现有详情页继续增强，加入“用户常见说法”“我们会怎么判断”“适合免费先问的情况”。
- `/knowledge` 或 `/help`：小酷知识库式帮助页，承载所有 FAQ 和服务规则。

### 5.2 小纸条 2.0

小纸条不只是表单，而是需求收集向导。

核心交互：

- 用户可以先写很模糊的一句话。
- 点击“让 AI 帮我整理”后，系统生成更清楚的版本，但保留原话。
- AI 同时给出缺失项提醒，如“补充截止时间”“上传源文件”“说明是否需要保留格式”。
- 用户可以上传 Excel、Word、PDF、PPT、图片和截图。
- 表单展示“这只是先提交看看，不一定收费”的提示。
- 用户可以选择“只是咨询/先判断”“希望尽快报价”“已经明确要做”三种意图。

数据字段建议：

- `original_demand`：用户原话。
- `polished_demand` 或继续使用 `demand`：AI 润色后的需求。
- `intent`：`consultation`、`quote_request`、`ready_to_start`。
- `missing_fields`：AI 判断缺失项。
- `service_confidence`：AI 判断服务分类的置信度。
- `customer_expectation`：用户期望结果。

### 5.3 管理后台 2.0

后台从列表 + 编辑表单升级为订单操作台。

必须新增：

- 搜索栏：支持订单号、客户邮箱/联系方式、服务类型、需求关键词、附件名、状态、标签搜索。
- 快捷筛选：未读、待追问、待报价、待收定金、处理中、待验收、待尾款、已完成、咨询类。
- AI 订单摘要：展示用户原话、AI 整理、标签、风险、建议追问、建议状态。
- 自动待办：每个订单生成下一步建议，如“追问截止时间”“发送报价”“上传交付物”“提醒验收”。
- 客户沟通草稿：AI 根据订单上下文生成管理员可编辑的回复，不自动发送。
- 进度建议：AI + 规则给出建议状态，管理员一键采纳。
- 异常提醒：长期未更新、已报价未付款、已交付未验收、用户补充新附件等。

后台页面结构建议：

- 左侧：订单搜索、筛选、标签、列表。
- 中间：订单详情、时间线、原话、附件、报价/付款/交付物。
- 右侧：AI 操作面板，包括摘要、待办、建议追问、状态建议、回复草稿。

### 5.4 订单进度自动维护

2.0 需要一个 `order_automation` 模块，输入订单事件流，输出建议动作。

事件来源：

- 用户提交小纸条。
- 用户补充消息。
- 用户上传附件。
- AI 润色完成。
- 管理员发送报价。
- 管理员记录定金/尾款。
- 管理员上传交付物。
- 用户点击验收。
- 订单长时间无动作。

自动规则：

- 新订单：自动 `submitted`。
- 用户表达“先问问/咨询/能不能”：标记 `consultation`，建议 `clarifying`，不建议立即报价。
- 用户上传附件：生成“检查附件并确认材料是否齐全”的待办。
- 管理员发送报价：可自动进入 `quoted`。
- 报价包含定金且未收：建议 `deposit_pending`。
- 定金已收：建议 `in_progress`。
- 上传交付物：建议或自动进入 `review`。
- 用户验收：自动进入 `completed`。
- 超过设定时间无动作：生成提醒，但不自动催促客户，先提示管理员。

AI 输出必须分级：

- `info`：普通摘要或提醒。
- `suggestion`：建议管理员操作。
- `safe_auto`：可自动执行的低风险状态更新。
- `requires_approval`：必须管理员确认。

### 5.5 小酷 Agent

小酷不是普通聊天框，而是“会动、会引导、会解释业务”的虚拟服务助手。

#### 人设

- 名字：小酷。
- 性格：机灵、轻松、靠谱，像一个守在窗口边的小助手。
- 语气：不端着，不承诺过度，常说“先给我看看”“这个可以先判断”“这里可能要补一张截图”。
- 边界：不承诺价格和成功率，不替管理员做最终报价，不要求用户泄露敏感密码。

#### 形象与动作

2.0 可先用轻量 SVG/Lottie/CSS sprite，不急着上 3D。

基础动作：

- idle：眨眼、轻微漂浮、看向用户。
- follow：靠近鼠标方向移动，但保持距离，不遮挡按钮。
- curious：用户停留在服务卡片时，小酷探头。
- thinking：AI 润色或回答时转圈/翻小本子。
- happy：用户提交成功时开心跳一下。
- alert：用户表单缺字段时轻轻提醒。
- sleep：用户长时间不动时打盹。
- hide：用户关闭或滚动到密集表格时缩到角落。

交互规则：

- 小酷默认出现在右下角，不挡主 CTA。
- 鼠标移动时，小酷只做“追随趋势”，不逐像素追，避免烦。
- 移动端默认折叠成小头像，点击展开。
- 提供“别跟着我”“隐藏本页”“减少动画”三个控制。
- 对话面板支持服务问答、帮我选服务、帮我写小纸条、查看订单状态。

#### 小酷知识库

小酷的知识库来源应统一：

- `serviceCatalog`：服务内容。
- FAQ：付款、交付、风险、范围。
- 订单规则：状态含义、咨询不收费、验收/定金/尾款说明。
- 当前页面上下文：用户在哪个服务页、是否登录、是否有订单。
- 当前用户自己的订单摘要：只在登录后可读取自己的订单，不能读取其他用户或内部字段。

小酷可回答：

- “你们能帮我做什么？”
- “PDF 翻译保排版能不能做？”
- “我不知道怎么分类怎么办？”
- “这个要不要收费？”
- “我该上传什么材料？”
- “我的订单现在到哪一步了？”

小酷不能回答：

- 内部成本、利润、管理员备注。
- 其他用户订单。
- 具体保证型承诺，如“一定开通成功”“一定当天完成”。
- 敏感凭证请求，如“把密码发给我”。

### 5.6 对话设计

小酷对话不是通用闲聊，而是业务导向。

对话模式：

- 服务咨询：基于知识库回答。
- 需求澄清：问 1 个最关键问题，不连续轰炸。
- 表单协助：把对话结果填入小纸条草稿。
- 订单说明：解释订单状态和下一步。
- 人工转接：生成给管理员看的摘要，而不是假装真人已介入。

对话输出结构：

- 简短回答。
- 推荐下一步。
- 可点击动作按钮，如“帮我写小纸条”“去文档处理服务页”“上传文件”“查看订单”。

## 6. 权限与安全

1. 小酷未登录时只能回答公共知识库和引导写小纸条。
2. 小酷登录后只能读取当前用户自己的订单摘要。
3. 小酷不能读取管理员字段：成本、利润、内部备注、内部优先级。
4. 管理员 AI 面板可以读取内部字段，但所有 AI 建议都要保留审计日志。
5. 用户上传文件默认只保存 metadata 和对象 key；文件解析能力需要单独加权限提示。
6. AI 生成的客户回复必须由管理员点击发送，不能自动发。
7. 订单状态自动更新必须区分低风险和高风险动作。

## 7. 体验细节建议

### 面向客户

- 服务页增加“你可能会这样说”示例，降低表达门槛。
- 小纸条成功后，小酷出现并解释“这只是先提交看看，不一定收费”。
- 订单页用小酷解释每个状态，而不是只显示状态标签。
- 如果用户上传了附件，小酷提示“已收到文件，酷里会先判断是否足够”。

### 面向管理员

- 后台默认按“需要处理”排序，而不是按更新时间裸排。
- 新订单出现时列表有轻量高亮。
- 搜索支持中文分词不必一步到位，2.0 可以先做大小写不敏感的关键词匹配。
- AI 面板展示“为什么这么建议”，避免黑箱。
- 每条建议都有“采纳 / 忽略 / 稍后提醒”。

## 8. 分阶段路线

### Phase 1：后台搜索与自动化基础

- 后台订单搜索栏。
- 订单 intent 字段。
- 订单待办和自动化建议表。
- 基于规则的状态建议。
- 管理员 AI 面板第一版。

### Phase 2：小酷 MVP

- 小酷浮动头像和动作状态。
- 小酷聊天面板。
- 知识库问答基于服务目录和 FAQ。
- 小酷可引导写小纸条。
- 可关闭、减少动画、移动端折叠。

### Phase 3：订单进度自动维护

- 事件流驱动自动化建议。
- 低风险状态自动更新。
- 高风险动作管理员确认。
- 客户沟通草稿。
- 超时提醒和未读提醒。

### Phase 4：服务知识库产品化

- 后台维护知识库。
- 服务页和小酷共用知识源。
- 业务规则版本化。
- 后续接真实 LLM / RAG / 文件解析。

## 10. 验收标准

1. 管理员能在后台搜索订单号、客户、关键词、附件名和服务类型。
2. 新小纸条提交后，管理员后台在刷新或轮询后能看到新订单，并显示 AI 摘要、原话和建议下一步。
3. AI 能根据订单内容生成待办、建议追问和建议状态。
4. 报价、付款、交付、验收等动作能触发自动化建议或安全状态更新。
5. 用户页面显示小酷，且小酷有 idle、follow、thinking、happy、hide 等基础动作。
6. 小酷能回答服务内容、收费边界、材料要求和订单状态含义。
7. 小酷不能暴露内部字段或其他用户订单。
8. 用户可以关闭小酷或减少动画。
9. 服务详情页内容与小酷知识库回答一致。
10. 所有 AI 建议保留可审计记录。

## 11. 额外建议

1. 增加“服务适配度评分”：用户写完小纸条后，系统提示“材料完整度 60%，建议补充截止时间和源文件”。
2. 增加“订单健康度”：后台显示每单是否卡在等待客户、等待管理员、等待付款、等待验收。
3. 增加“报价模板”：管理员选择服务类型后自动生成报价结构，但价格仍由人确认。
4. 增加“客户可见下一步”：用户订单页明确显示“现在等谁做什么”。
5. 增加“小酷情绪与状态联动”：订单提交成功开心，等待时安静，报错时提醒，不一直打扰。
6. 增加“知识库一致性测试”：服务目录、FAQ、小酷回答素材必须来自同一数据源，避免页面和 agent 说法不一致。

## 12. 当前实现影响点

需要重点改造的现有模块：

- `backend/src/catalog.ts`：升级为知识库源或迁移到 `knowledge` 模块。
- `backend/src/agent.ts`：从规则摘要扩展为订单自动化建议引擎。
- `backend/src/database.ts`：新增 intent、next action、automation suggestions、todos、agent sessions。
- `backend/src/app.ts`：新增后台搜索、agent chat、automation API。
- `frontend/src/App.tsx`：需要拆分页面和组件，避免继续变大。
- `frontend/src/api.ts`：新增搜索、agent chat、automation endpoints。
- `frontend/src/styles.css`：新增小酷动画、后台 AI 面板、搜索筛选样式；后续建议拆分 CSS。

## 13. 建议的技术原则

1. 自动化先做可解释，再做自动执行。
2. 小酷先做轻量 2D 动画，不先上复杂 3D。
3. 所有知识内容从同一份结构化数据生成。
4. 客户侧 agent 和管理员侧 agent 分权设计。
5. 订单状态变更必须由事件流驱动，不能只改一个字段。
6. 高风险 AI 动作必须管理员确认。
7. 每个 2.0 功能都要能拆成独立 GitHub issue 和独立验收。
